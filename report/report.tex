\Abstract{
We are developing an AI assistant for the UL FRI student office that answers administrative questions in Slovenian and English based on official faculty documents. At this stage, we implemented an end-to-end pipeline for (i) collecting public UL/FRI web content, (ii) parsing heterogeneous documents (HTML, PDF, DOCX) into structured text, (iii) section-aware chunking, (iv) multilingual embedding and FAISS indexing, and (v) retrieval and optional answer generation via instruction-tuned LLMs. The system supports dense retrieval, optional BM25+dense hybrid retrieval, and optional cross-encoder reranking. For evaluation, we maintain a JSONL question set whose questions are annotated with the relevant chunk identifiers, enabling standard information retrieval metrics (Recall@k, Precision@k, MRR@k, nDCG@k) and basic end-to-end QA proxies such as keyword hit rate, together with retrieval and generation latency measurements.
}

\section*{Methods}

\subsection*{Pipeline overview}
	At this stage, our implementation provides a complete data-to-index-to-answer pipeline, together with scripts for retrieval testing and evaluation. The workflow is run-based: each run stores its raw downloads, processed JSONL files (parsed documents and chunks), and the final FAISS index.

\subsection*{Data collection}
	We collect publicly available content from UL FRI and other UL domains using a lightweight crawler that starts from a curated list of seed URLs. The crawler performs breadth first traversal with domain-specific depth limits (separately for \texttt{fri.uni-lj.si}, other \texttt{*.uni-lj.si} domains, and external domains). It downloads HTML pages as well as linked PDF/DOCX attachments; discovered attachments do not increase the crawl depth. Each downloaded file is stored under the run's \texttt{raw/} directory using a stable URL-derived path, and a \texttt{manifest.jsonl} file stores provenance (source URL, download time, content type, and SHA-256 hash).
	We also discover URLs from sitemap files and RSS/Atom feeds (with configurable caps), respect robots.txt when available, and run with a fixed delay between requests. Update mode crawls use conditional requests (ETag / If-Modified-Since) to avoid redownloading unchanged content, while the run snapshots the seed list and crawl parameters for reproducibility. Domain rules further restrict binary downloads to UL domains only and apply a stricter policy for \texttt{ucilnica.fri.uni-lj.si} (HTML-only, Slovene/English pages, no binary attachments).

\subsection*{Document parsing and normalization}
	Downloaded documents are parsed into a common JSON structure consisting of a document identifier (\texttt{doc\_id}), title, language, and a list of named sections. We support:
	\begin{itemize}[noitemsep]
		\item \textbf{HTML:} parsed with BeautifulSoup (optionally trafilatura extraction when available); the title is extracted from standard page elements, and content is split into sections based on heading tags. We also extract publish/modified dates from meta tags, \texttt{<time>} elements, and JSON-LD when present.
		\item \textbf{PDF:} parsed with Docling when available (with OCR support for scanned documents; Tesseract CLI is used on CPU when needed); otherwise PyMuPDF is used as a fallback. We also parse PDF creation/modification timestamps.
		\item \textbf{DOCX:} parsed with \texttt{python-docx}, preserving heading structure and converting tables into text rows; document core properties (created/modified) are captured when available.
		\item \textbf{Images:} indexed via required sidecar \texttt{.txt}/\texttt{.md} descriptions (the image itself is not OCR'd).
	\end{itemize}
	To keep identifiers stable across machines and re-downloads, \texttt{doc\_id} is derived preferentially from the canonical URL; if no URL is known, we fall back to content hashes. We also perform basic whitespace normalization and detect language (Slovenian/English) to drive prompting. The crawler manifest is merged into document metadata (URL, download time, HTTP last-modified/date, SHA-256) to preserve provenance and enable recency-aware retrieval, and unsupported languages can be filtered at parse time.

\subsection*{Chunking}
	Documents are chunked with a section-aware sliding window. Chunk size and overlap are measured in whitespace-split tokens (default: 400-token chunks with 80-token overlap; tuning results are reported in Results). Each chunk is contextualized by prefixing the document title and section name (when available), and receives a stable \texttt{chunk\_id} of the form \texttt{<doc\_id>\_chunkXXXX}.
	We additionally deduplicate near-duplicate chunks with a SimHash-based filter (configurable distance/banding) and drop noisy fragments to reduce repeated boilerplate in the index.

\subsection*{Embeddings and vector index}
	For semantic search we embed chunks with the multilingual E5 sentence-transformer (\texttt{intfloat/multilingual-e5-base}). We apply E5-style prefixes (\texttt{passage:} for chunks and \texttt{query:} for questions) and L2-normalize embeddings. The index is built with FAISS \texttt{IndexFlatIP} (inner product); due to normalization this is equivalent to cosine similarity. For reproducibility, we persist the FAISS index together with chunk metadata (text, document title/section, URL, and scores).
	The indexer also writes a compact sources list for each run (URL, local path, hashes, timestamps), and snapshots sidecar image descriptions so that runs remain self-contained even if the raw dataset changes later.

\subsection*{Retrieval and reranking}
	Given a question, we compute its embedding and retrieve the top-$k$ chunks from FAISS. The implementation also supports:
	\begin{itemize}[noitemsep]
		\item \textbf{Hybrid retrieval:} BM25 lexical scores (Rank-BM25) are fused with dense scores on a candidate pool (union of lexical and dense top results), after min--max normalization of each score component.
		\item \textbf{Cross-encoder reranking:} an optional sentence-transformers CrossEncoder reranks the retrieved candidate set (default candidate pool size 20) and returns the final top-$k$ list.
	\end{itemize}
	We additionally compute a simple \texttt{retrieval\_weak} flag based on a score threshold; this is used both in the UI and in prompting as a warning signal when retrieval confidence is low. Retrieval can filter by language and allowed domains, and we apply a lightweight recency bias to scores using document timestamps (published/created/modified/HTTP last-modified). The boost is additive and decays exponentially with a configurable half-life (default weight $0.08$, half-life $365$ days), so more recent sources are slightly favored without overriding semantic relevance.

\subsection*{Prompting and answer generation}
	For end-to-end question answering, retrieved chunks are formatted into a context block with numbered sources. A language-specific system prompt instructs the model to answer \emph{only} from the provided context and to respond in the detected language (Slovenian or English). Generation uses Hugging Face Transformers causal language models with configurable decoding parameters (maximum new tokens, temperature, and nucleus sampling). The system can also run in retrieval-only mode, which is useful for debugging and evaluation without requiring GPU resources.

\subsection*{Evaluation setup}
	We evaluate on a JSONL file of manually written questions. Each question includes the expected language and expected keywords. Importantly, we maintain \texttt{relevant\_chunk\_ids} (and optionally \texttt{relevant\_doc\_ids}) for each positive question, i.e., the questions have already been indexed to the gold chunks containing the answer. This enables reporting standard retrieval metrics at a chosen $k$ (chunk-level Recall@k, Precision@k, MRR@k, and nDCG@k). For end-to-end QA we additionally report keyword hit rate in generated answers, as well as retrieval and generation time per question. We also include negative (out-of-domain) questions to measure abstention behaviour via the \texttt{retrieval\_weak} heuristic.
	To ensure chunk-level metrics stay valid across different chunk sizes, we keep a global question set without chunk IDs and create per-run question files, then refresh chunk labels per run before evaluation. We use retrieval-only evaluation (\texttt{scripts/evaluate.py --retrieval-only}) for fast tuning of retrieval settings and chunk sizes, and a small manual retrieval script (\texttt{scripts/test\_retrieval.py}) for spot checks.

\subsection*{Chunk size tuning}
We tuned chunk size on the Test run using retrieval-only evaluation with hybrid retrieval + reranking at $k=6$ (chunk overlap 80). The evaluation set contains 61 questions (52 Slovene, 9 English), with 47 annotated positives and 10 negative questions. Table~\ref{tab:chunk_tuning} reports chunk-level metrics.

\begin{table*}[hbt]
	\caption{Chunk size tuning results (chunk-level, retrieval-only, hybrid + rerank, $k=6$).}
	\centering
	\begin{tabular}{r|c|c|c|c|c}
		\toprule
		\textbf{Chunk size} & \textbf{Recall@6} & \textbf{Precision@6} & \textbf{MRR@6} & \textbf{nDCG@6} & \textbf{Avg. time [s]} \\
		\midrule
	100 & 89.36\% & 21.28\% & 80.32\% & 77.36\% & 1.36 \\
	200 & 97.87\% & 24.11\% & 88.65\% & 86.73\% & 1.67 \\
	300 & 97.87\% & 24.47\% & 90.07\% & 87.66\% & 2.00 \\
	400 & 97.87\% & 24.11\% & 90.43\% & 88.51\% & 1.94 \\
	500 & 97.87\% & 23.76\% & 91.13\% & 88.81\% & 1.91 \\
		\bottomrule

Overall, chunk sizes 200--500 achieve similarly high recall, while 100 underperforms. We select 300 tokens as the default because it provides the best precision with strong ranking quality, while 500 yields slightly higher nDCG/MRR at lower precision.
