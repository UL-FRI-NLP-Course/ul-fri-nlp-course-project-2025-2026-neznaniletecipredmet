\Abstract{
We built an AI assistant for the UL FRI student office that answers administrative questions in Slovenian and English. The system is a RAG pipeline that crawls and parses UL/FRI web content, chunks it with a section-aware splitter, embeds the chunks with a multilingual sentence-transformer, and stores them in FAISS. At query time we compare three retrieval modes -- dense, BM25+dense hybrid, and hybrid with a cross-encoder reranker -- feeding an instruction-tuned generator. We evaluate the pipeline with both classical retrieval metrics (Recall@$k$, MRR@$k$, nDCG@$k$) and a RAGAS-style LLM-as-judge protocol scoring faithfulness, hallucination, refusal handling, and overall quality. Early on we noticed that many of our reference answers were not actually grounded in the corpus, so we first ran a Sonnet-graded audit and used the cleaned set for every comparison. An embedder $\times$ retrieval-mode sweep followed by a chunk-size sweep picks \texttt{multilingual-e5-base} with hybrid retrieval, a cross-encoder reranker, top-$k{=}4$, and 400-token chunks as the best configuration, with a Sonnet-graded in-scope faithfulness of $0.740$ and $11/11$ correct refusals on out-of-corpus questions.
}

\section*{Introduction}
We are building an AI assistant for the student office of the Faculty of Computer and Information Science (UL FRI). The goal is a chatbot that answers everyday administrative questions about study programmes, regulations, and the office's work in Slovenian and English. Plain LLMs are a poor fit here: the answers depend on small details (deadlines, paragraph numbers, application forms) that need to come from the official documents, not from the model's training data. We therefore build the assistant on Retrieval Augmented Generation \cite{lewis2020retrieval}, which has been a good fit for university-style FAQ chatbots in prior work \cite{ranoliya2017chatbot,adamopoulou2020overview} and which a recent survey shows helps with hallucination and out-of-date knowledge in educational deployments \cite{SurveyRAGeducation}. Our pipeline borrows the comparative-retrieval setup (dense vs.\ hybrid vs.\ reranked, with a chunk-size sweep) from RAG-for-admissions and university-resources studies \cite{llmStudentAdmissions,mississippiUniversity2024}, and the choice to evaluate locally hosted instruction-tuned LLMs from work on open-day chatbots \cite{universityOpenDaysLocalLLMs} and on RAG vs.\ fine-tuning for knowledge-intensive tasks \cite{ovadia2024finetuningretrievalcomparingknowledge}.

\section*{Methods}

\subsection*{Data collection and parsing}
The pipeline is organised as \emph{runs}: each run lives in its own directory holding the raw downloads, parsed JSONL documents, chunks, and the FAISS index, so we can rebuild any single stage without touching the others. A lightweight crawler starts from a curated seed list and walks breadth-first, with separate depth limits for \texttt{fri.uni-lj.si}, the other \texttt{*.uni-lj.si} domains, and external domains. It downloads HTML pages along with their linked PDF/DOCX attachments, augments the crawl with sitemaps and RSS/Atom feeds, respects \texttt{robots.txt}, throttles requests, and stores SHA-256 hashes plus HTTP timestamps in a per-run \texttt{manifest.jsonl}; update crawls reuse \texttt{ETag}/\texttt{If-Modified-Since}. We then parse each document into a common JSON record (stable URL-derived \texttt{doc\_id}, title, language, ordered named sections): HTML via BeautifulSoup (with optional trafilatura), PDF via Docling with Tesseract OCR or PyMuPDF as a fallback, DOCX via \texttt{python-docx}. Image-only sources are indexed via required sidecar text descriptions.

\subsection*{Chunking, embeddings and retrieval}
We split documents with a section-aware sliding window measured in whitespace tokens (default $400$, overlap $80$). Every chunk is prefixed with the document title and section name, gets a stable \texttt{chunk\_id}, and is then deduplicated with a SimHash filter. The chunks are embedded with \texttt{intfloat/multilingual-e5-base} -- using the E5-style \texttt{passage:}/\texttt{query:} prefixes and L2 normalisation -- and indexed in FAISS as an \texttt{IndexFlatIP} (which is cosine under normalisation). At query time we support three retrieval modes: dense (top-$k$ from FAISS), hybrid (BM25 fused with dense scores after min--max normalisation on the union of both top lists), and hybrid plus cross-encoder rerank (a sentence-transformers cross-encoder reranks a candidate pool of size $20$). Retrieved chunks are then formatted into a numbered context block and a language-specific system prompt tells the generator to answer \emph{only} from that block and in the detected language.

\subsection*{Evaluation set and reference audit}
Our evaluation set is a hand-written JSONL file of $61$ questions ($51$ \emph{in-scope} -- the corpus contains a factual answer -- and $10$ \emph{negatives}, deliberate out-of-corpus probes used to test refusal). Each entry carries the expected language and a list of expected keywords; positives also carry \texttt{relevant\_chunk\_ids} for chunk-level retrieval metrics and a free-form \texttt{reference\_answer} that we use as the human-written gold for the LLM-as-judge protocol below.

The first time we ran the judge, in-scope faithfulness was very low. Inspection showed many of our reference answers contained details the corpus actually does not support, so the judge was, correctly, marking them down. To stop this reference drift from contaminating later comparisons we run a Sonnet-graded fact-check before the cross-configuration sweeps: for each positive question we retrieve a top-30 BM25 $\cup$ dense pool from the current index and ask Claude Sonnet (\texttt{claude-sonnet-4-6}, $T{=}0$) to label every claim in the reference as \texttt{supported}, \texttt{partial}, or \texttt{unsupported}, and to optionally produce a grounded rewrite. The application policy is fixed: \texttt{supported} keeps the original; \texttt{partial} or \texttt{unsupported} with a rewrite uses the rewrite (flagged \texttt{auto\_rewritten}); \texttt{unsupported} without a rewrite is dropped; negatives are never modified. The cleaned set is written as a versioned artefact \texttt{questions\_full\_v2.json}.

\subsection*{LLM-as-judge protocol}
We use a RAGAS-style judge: for each row we send the question, the retrieved top-$k$ chunks, the model answer, and the reference answer to Claude with a strict-JSON rubric covering \emph{faithfulness}, \emph{answer relevance}, and \emph{context relevance} (each in $[0,1]$), binary \emph{hallucination} (per-answer flag, set when at least one claim is unsupported by the retrieved context) and \emph{refusal} flags, and an integer \emph{overall} score in $[1,5]$. Negatives are scored under the same rubric, with a correct refusal rewarded with high faithfulness. All judge calls run at $T{=}0$ for reproducibility. To keep cost down without sacrificing the bottleneck quality we use Sonnet (\texttt{claude-sonnet-4-6}) for the reference audit and the final re-judgment, and Haiku (\texttt{claude-haiku-4-5}) for the comparison sweeps -- the two judges agreed to within $0.001$ in in-scope faithfulness on the winning configuration ($0.739$ vs.\ $0.740$), so we trust Haiku for ranking. Per-configuration indices are built by short driver scripts that symlink \texttt{raw/} from the source crawl and pass \texttt{EMBEDDING\_MODEL}/\texttt{CHUNK\_SIZE}/\texttt{CHUNK\_OVERLAP} as environment overrides; the chosen parameters are persisted to \texttt{embedding\_info.json} next to the index and re-checked on load, so we don't accidentally read an index with a different embedder or chunk size.

\section*{Results}

\subsection*{Corpus, evaluation set, and reference audit}
At the default $400/80$ chunk policy the full crawl yields $2{,}742$ parsed documents and $29{,}296$ chunks, up from the $176$ documents in the initial \emph{Test} corpus we used during development. Building the index for the full crawl took roughly $4$ hours when OCR was on; disabling OCR (PyMuPDF text extraction only) brought that down to $\sim14$ minutes for the same $\sim500$ PDFs and we did not see any measurable retrieval-quality drop afterwards, so we kept it off for all later experiments. Running the Sonnet audit on the original references against this corpus was a wake-up call: only $9/47$ ($\sim19\%$) of the in-scope references were fully supported as written. Of the rest, $31$ were classified \texttt{partial} and rewritten using only chunk-attested facts, $7$ were \texttt{unsupported} ($3$ dropped, $4$ rewritten where Sonnet found alternative chunks), and $14$ were skipped (no factual reference, or negative). The cleaned \texttt{questions\_full\_v2.json} contains $54$ in-scope and $11$ negatives and is the basis for everything that follows.

\subsection*{Chunk-level retrieval (Test corpus)}
On the Test corpus, where we have chunk-level annotations, dense retrieval at $k{=}4$ already reaches $88.0\%$ Recall@4 and $66.3\%$ nDCG@4. Adding the cross-encoder reranker lifts these to $100.0\%$ and $93.4\%$, while hybrid + rerank is essentially tied ($100.0\%$ / $93.9\%$). The corresponding chunk-size sweep on the same corpus showed sizes $200$--$500$ all reaching $\sim97.9\%$ Recall, with $300$ giving the highest precision; the LLM-judged sweep on the full crawl, reported below, revisits this choice on a stricter metric and ends up picking $400$.

\subsection*{Embedder $\times$ retrieval-mode sweep (LLM-as-judge)}
Table~\ref{tab:embedder_matrix} shows the embedder $\times$ retrieval-mode sweep at top-$k{=}4$ on the cleaned set, judged by Claude Haiku at $T{=}0$; refusal-side faithfulness is $\geq 0.90$ across all six configurations. Two things stood out for us. First, the cross-encoder reranker is by far the biggest lever: on \texttt{e5-base} it lifts faithfulness from $0.575$ (dense) and $0.565$ (hybrid) to $0.739$, and context relevance from about $0.72$ to $0.857$. Second, the result we did not expect -- \texttt{e5-base + rerank} beats \texttt{e5-large + rerank} on every in-scope metric while embedding about $2.5\times$ faster. Once the cross-encoder does the final ranking, the smaller first-stage embedder is enough.

\subsection*{Chunk size on the winning configuration}
Holding the winner fixed (\texttt{e5-base}, hybrid + rerank, $k{=}4$) and varying the chunk size with $20\%$ overlap, the Haiku-judged in-scope faithfulness is $0.691$ at $200/40$ ($50{,}546$ chunks), $0.698$ at $300/60$ ($36{,}331$), and $0.739$ at $400/80$ ($29{,}296$). Larger chunks help here because the cross-encoder benefits from a bit more local context per chunk, and $400$-token chunks also produce the smallest, fastest index of the three -- so we keep them as the default.

\subsection*{Headline configuration and ablation}
We then re-judged the winning configuration with Claude Sonnet at $T{=}0$ on the cleaned set (Table~\ref{tab:headline}). The headline in-scope faithfulness is $0.740$, answer relevance $0.820$, context relevance $0.807$, overall $3.605/5$, and hallucination rate $44.2\%$; on the $11$ negatives the system refuses correctly in all $11/11$ cases with refusal-side faithfulness $0.905$. Walking the gain through the stages: the original baseline (small corpus, noisy refs, dense $k{=}8$) sat at Haiku-judged faithfulness of $0.34$; the full crawl plus reference audit at dense $k{=}4$ already moves it to $0.575$ ($+0.24$); hybrid on its own does not help ($0.565$); adding the cross-encoder reranker takes it to $0.739$ Haiku / $0.740$ Sonnet ($+0.16$), with the per-answer hallucination rate dropping from $80.9\%$ to $44.2\%$.

\section*{Discussion}

Both the retrieval-only and the LLM-as-judge results point at the same story. The biggest improvements split roughly evenly between two stages: corpus expansion and the Sonnet audit together move dense-$k{=}4$ faithfulness from $0.34$ to $0.575$ (about $58\%$ of the total gain), and the cross-encoder reranker takes it from there to $0.740$ (the remaining $42\%$). Hybrid retrieval on its own does almost nothing, which surprised us; it only starts to pay off once the reranker is on top of it, presumably because the union of BM25 and dense candidates gives the cross-encoder a more diverse pool to choose from. Another finding we did not expect was that \texttt{multilingual-e5-base} beats \texttt{multilingual-e5-large} once reranking is in play while embedding $\sim2.5\times$ faster: the cross-encoder absorbs the work that would otherwise depend on a stronger first-stage embedder.

The biggest caveats are around the evaluation set: $54$ in-scope and $11$ negative questions are not many, the language is mostly Slovenian, and even with $T{=}0$ a single LLM judge has some per-question variability -- a stratified manual review with Cohen's $\kappa$ against the Sonnet judge is the next thing on our list. The headline numbers are reported with a small Qwen generator that fits on a laptop; we have wired up an end-to-end comparison against \texttt{GaMS3-12B}, \texttt{Llama-3.1-8B}, and \texttt{Mistral-7B} on the cluster (same retrieval configuration, same judge), and a $3$-question smoke test verified the flow, but the cluster jobs were not finished by the time of this submission. We also disabled OCR to keep full-crawl build time tractable: for scanned PDFs this would silently drop content, so a future iteration should re-run Docling/Tesseract just on the subset where PyMuPDF returns near-empty text.
